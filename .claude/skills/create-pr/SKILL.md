# create-pr

## Overview

Walks through creating a MiraNote-AI pull request that passes the project's enforced rules on the first try. The skill uses the same regex/whitelist logic the CI runs (`checks/pr_has_reference.py` for body, `checks/pr_title_format.py` for title), so violations get caught locally instead of in CI feedback loops.

## When to use

- Before running `gh pr create` in any of `miranote-api`, `miranote-ios`, `miranote-web`, `mirabot`, `.github`.
- Drafting a PR title/body and unsure whether it complies with project rules.
- A PR was just opened and CI's `theta` or `zeta` stage failed — use this skill to fix and re-push.

Do not use:
- For sync-bot PRs (head branch matches `chore/sync-ai-docs-*`) — Rules 6 and 8 exempt these.
- For local commits without a PR — Rules 6 / 8 apply to PRs only.

## Quick reference

### Title format (Rule 8, enforced by theta)

```
<type>(<scope>)?(!)?: <description>
```

| Field | Allowed values |
|---|---|
| `<type>` | `feat` / `fix` / `chore` / `docs` / `refactor` / `test` / `ci` / `perf` / `build` / `revert` |
| `<scope>` (optional) | `api` / `web` / `ios` / `bot` / `infra` |
| `!` (optional) | mark breaking change |
| `<description>` | imperative mood; lowercase first letter; no trailing period; no `#N`; no `WIP`/`DRAFT`/`FIXME`/`TODO`; total title <= 72 chars |

Good:
- `feat(api): add user signup endpoint`
- `fix: handle empty request body in parser`
- `ci: remove admin bypass for main branch protection`

Bad:
- `ci(F-final): enforce_admins=true` — scope not in whitelist
- `feat: Added login` — past tense + capital first letter
- `fix: broken thing #123` — issue ref in title
- `chore: WIP cleanup` — WIP marker

### Body (Rule 6, enforced by zeta)

PR body must contain at least one of:
- `#<int>` — issue or PR number (recommended; use `create-ticket` skill first to file the issue)
- a URL (`http://` or `https://`)
- `spec:` / `design:` / `adr:` / `rfc:` followed by a token

### Soft rules (not enforced; expected by convention)

- One PR does one thing. If the title needs "and" or multiple commas, split the PR.
- Avoid vague verbs. `fix: bug` -> `fix: off-by-one in pagination cursor`.
- No backticks in title (they render poorly in notification emails / Slack).
- No "by Claude" / "generated by AI" markers — they don't add user value.

## Workflow

1. **Branch name:** `<type>/<short-slug>`, e.g. `feat/user-signup`. Do not embed internal indices (`f-2`, `phase-3`).
2. **Draft the title** matching the format above.
3. **Draft the body** with at least one valid reference.
4. **Pre-check locally** before `gh pr create`. From a clone of `MiraNote-AI/.github`:

   ```bash
   cd path/to/MiraNote-AI/.github
   PR_TITLE="<your title>" PR_BRANCH="<your branch>" \
     PYTHONPATH=. python3 -m checks.pr_title_format
   PR_BODY="<your body>" PR_BRANCH="<your branch>" \
     PYTHONPATH=. python3 -m checks.pr_has_reference
   ```

   Both must exit 0. If either prints errors, fix the title/body and re-run.

5. **Create the PR:** `gh pr create --title "<title>" --body "<body>"`.
6. **Confirm** the `theta` and `zeta` stages pass in CI.

A Discord announcement to `#general` is sent automatically by the
`pr-discord-announce.yml` workflow on PR open, posting as the team bot
(mirabot/LumaBot). **Do not** send team-facing announcements through your
local `miranote-discord` MCP server -- that connection runs under a personal
bot identity (Meng-Proxy), not the team bot, and using it for team comms
mixes identities in the channel. The MCP server is for personal read-only
investigation (looking up messages, finding members), not for posting on
behalf of the team.

## Common mistakes

| Mistake | Fix |
|---|---|
| Scope is an internal codename (`F-2`, `phase-3`) | Use real component (`api`/`web`/`ios`/`bot`/`infra`) or omit |
| Past tense / gerund verb (`added`, `fixing`) | Use imperative: `add`, `fix` |
| `#123` in title | Move to body, e.g. `Closes #123` |
| `WIP` in title | Use GitHub's Draft PR state |
| Title ends with `.` | Drop the trailing period |
| Body has no reference | Add `Closes #N`, a design doc URL, or `spec:`/`design:`/`adr:`/`rfc:` token |
| Title > 72 chars | Move details to body; keep title scannable |

## Red flags

- Title starts with a capital letter after the colon
- Title has parentheses with anything other than `api` / `web` / `ios` / `bot` / `infra`
- Title contains `#`, `WIP`, `TODO`, `DRAFT`, or `FIXME`
- Body is just a one-liner with no link

Any of these = stop, fix, re-run the pre-check.
